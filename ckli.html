CKli

CKli est un outil agrégeant des dépôts de code et automatisant des actions sur ces agrégats.
Il a été initié par Olivier Spinelli, car il passait trop de temps à mettre à jour les outils de Signature Code.

Problématiques

Chaque librairie de Signature-Code est dans son propre dépôt de code.
Cela engendre deux problèmes :
Il est difficile de se représenter l’état général de toutes les librairies, et faire des opérations communes sur toutes ces librairies prend beaucoup de temps.

On appelle cet ensemble de librairies une « stack », celle de Signature-Code possède 38 dépôts de code, qui génère un total de 151 artefacts.
La moindre petite action à faire sur l’ensemble des dépôts engendre au moins une demi-journée de travail.

Solutions existantes

Il existe des méthodes et des solutions sur le marché, mais elles ne peuvent pas s’appliquer aux besoins de Signature-Code.
• La méthode du Monorepo : le « monolithic repository », ou dépôt de code monolithique en français, consiste à mettre toutes ses librairies dans le même dépôt de code.
Cela permet d’effectuer quelques types d’actions sur l’ensemble des projets, comme compiler, mais ne permet pas la gestion fine des droits d’accès : on ne peut pas donner l’accès à une personne à une seule librairie.
Or, c’est une fonctionnalité qu’on souhaite garder, on ne peut donc pas utiliser un Monorepo.
L’approche que nous utilisons, est appelée « multi-repo » pour multi repositories.
• Parmi les outils existants qui essayent d’aider à gérer les multi-repo, aucun n’est aussi poussé que ce que l’on veut.
La plupart se contentent de globaliser les opérations git, comme « Repo », l’outil utilisé pour gérer le millier de dépôts de code constituant Android.

Il nous fallait donc créer un outil pour cela, et cet outil s’appelle aujourd’hui CKli.
